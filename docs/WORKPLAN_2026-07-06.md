## Batch R — PDF reader
- **R1. Dim mode tuning** — make the Dim filter *slightly lighter + more yellowish* (warmer, less
  dimming). Current `sepia(0.35) brightness(0.93) contrast(0.95)` → warmer/lighter (e.g. raise sepia
  hue warmth, brightness ~0.96, add a faint amber tint). Tune by eye + re-capture.
- **R2. Dark mode bg** — the full invert makes the page near-black; change so the page background is
  a *yellowish dark grey* like the warm-dark theme surface (≈`#211e2a`/warm). Technique: instead of
  a pure `invert(1)`, reduce invert or composite the inverted text over a warm dark-grey backing
  (e.g. invert + slight `sepia`/`brightness` lift, or a canvas backing color) so the "paper" reads as
  warm dark-grey, not black. Tune + re-capture.
- **R3. Reference anchor buttons overhaul** — the rectangles drawn over references (anchor buttons)
  need to be more stable/robust (correct positioning across zoom/resize/re-render) AND **work in
  scroll (continuous) mode**, not just paged. Rework the overlay-box computation to be driven by the
  same per-page canvas geometry in both view modes; verify boxes track the text on zoom/resize.

## Batch P — Paper view / metadata
- **P1. Refresh consistency (broadened per owner 2026-07-06).** Two parts:
  (a) **Live-refresh the open paper view after a background job** (extract/enrich) produces new
  metadata — no navigate-away/back or reload (poll the work / subscribe to job completion for the
  open work id and refetch). Enrich may already partly work; cover extraction too.
  (b) **Mutations must refresh the relevant listing/view.** Actions like **delete** (and add/remove/
  move/rename) currently leave stale lists (e.g. a deleted paper stays in the library view). Audit
  the mutation handlers so each updates its view — prefer in-place list updates (already the pattern
  for status changes) or a targeted refetch — consistently across library, shelves, racks, tags,
  duplicates. Make "act → the view reflects it" the rule.
- **P2. Metadata-conflict "match %"** — for conflicting field values (esp. abstracts that differ only
  by formatting — end-of-line hyphenation, whitespace), compute and show a similarity/match
  percentage so the user can quickly judge how alike two values are. Use `rapidfuzz` (already a dep),
  normalizing whitespace/hyphenation first; show per-conflict.
- **P3. arXiv id / DOI not filled from web-fetched papers (bug)** — the identifier-import endpoint
  *does* set `arxiv_id`, so a different path drops it: trace the find-on-web / PDF-attach / enrichment
  path where a paper is fetched from arXiv (or its PDF parsed) but `arxiv_id`/`doi` stays empty, and
  persist the identifier (from the source URL/metadata, or parse from the PDF text). Investigate →
  fix the offending path(s).

## ★ Batch D — Duplicate resolution overhaul (needs owner confirm on semantics)
Current `merge_works` and `link_as_version` both just move file-links to the target (leaving one entry
with the PDF(s) and one empty) — misleading. **Proposed redesign:**
- **Merge** = *truly consolidate* two entries into one canonical paper: move ALL files, references,
  citation mentions, metadata assertions, tags, shelf/rack memberships, annotations, keywords from
  the source into the chosen target; fill the target's empty fields from the source (provenance
  kept); then the source **no longer appears in the library** — kept as a hidden redirect stub so
  existing references/links resolve to the target (recommended over hard delete, for undo/safety).
  Result: one complete paper, none left empty.
- **Link** = keep BOTH entries and their own PDFs; record a **bidirectional relationship**
  ("related / same work, different version") so from either paper you can see and jump to the other.
  No file movement, no deletion.
- Rename the actions accordingly and update the review UI + confirmations.
- **Owner confirm:** (1) this merge/link split; (2) merge source = hidden redirect stub (rec) vs hard
  delete. Awaiting go before implementing this batch.

## Batch W — Makefile & Docker workflow robustness (owner 2026-07-06)
Owner pulled, rebuilt, `make up-all` errored, had to hand-fix (commits `45edd6e`, `5f4e874`). Root
cause: the frontend `node_modules` **named volume goes stale when `package-lock.json` changes**
(new deps `yaml`/`echarts`/`cytoscape`) — the rebuilt image is shadowed by the old volume → missing
deps → runtime errors. Owner's fix (bake node_modules + lock-hash into the image at
`/opt/paracord-frontend`, self-heal in the entrypoint on hash mismatch, add `make up-all`, chown
`dist`) is correct. Now make the WHOLE workflow robust out-of-the-box AND after changes:
- **Verify + solidify the frontend self-heal**: empty-volume first-run (copies baked node_modules,
  no npm ci), hash-match (no-op), hash-mismatch-with-baked (fast copy), hash-mismatch-no-baked
  (npm ci fallback); confirm the `.paracord-package-lock.sha256` marker logic is airtight.
- **Audit every named volume for ownership**: confirm backend `/app/storage` (done), frontend
  `node_modules` + `dist` (done); check the **agent** container + any other writable mount and chown
  in its entrypoint if it runs non-root.
- **Startup ordering / self-heal**: confirm compose `depends_on` conditions (api waits for Postgres
  *healthy* before migrations; worker waits for migrations-at-head + Redis) so a cold `up-all` orders
  itself; api entrypoint runs `alembic upgrade head` (done) — make sure a fresh DB comes up clean.
- **Recovery + clarity Make targets**: add `make rebuild` (build `--no-cache` + recreate) and
  `make fresh`/`reset` (down + drop the app volumes — postgres/node_modules/dist/library — then
  build + up, for a guaranteed clean slate; keep ollama models), and a dev `make smoke`/`doctor`
  that after `up-all` asserts each service is healthy (api `/health`, frontend serving, worker up,
  grobid/ollama when profiled). Document the canonical flow (`make init && make up-all`) + recovery
  (`make fresh`) in INSTALL/a runbook.
- **Prove it**: exercise a rebuild cycle — simulate a `package-lock.json` change → `make up-all`
  (rebuild) → confirm the frontend picks up deps with NO manual fix; and a clean `up-all` → all
  services healthy (smoke passes). Report the evidence.
This runs right after Batch R (independent files — Makefile/Dockerfiles/entrypoints/compose/docs vs
