## ★ Batch D — Duplicate resolution overhaul (semantics CONFIRMED 2026-07-06)
Current `merge_works`/`link_as_version` both just move file-links (one entry ends full, one empty).
Confirmed redesign:
- **User-selectable merge order.** In the duplicate view, item #1 = **base** (merge INTO), item #2 =
  **merge-from**. A double-arrow / clicking swaps them, making #2 the base. The chosen base is the
  surviving canonical entry.
- **Merge** = consolidate into the base: **fill the base's empty fields** from the other; for fields
  with values in **both**, **add the other's value as a metadata *conflict*** on the base (reusing
  the existing per-field conflict/assertion mechanism) so the user resolves it manually. Move all
  files, references, citation mentions, tags, shelf/rack memberships, annotations, keywords into the
  base; **redirect** the source's incoming references to the base (the "link-fixing" needed either
  way). The source becomes a **hidden shadow** of the base (not deleted) → enables **Unmerge**.
- **Unmerge approach — RECOMMENDED: single-level undo + flatten-on-re-merge** (over a full merge-
  history stack). Rationale: a fully-reversible chained history must perfectly record + reverse
  every moved entity (files/refs/mentions/tags/shelves/annotations/filled-fields/added-conflicts) —
  genuinely bug-prone and exactly the "super complicated / stability-risky" thing to avoid. Instead:
  - The shadow stores a **merge-record** (what was filled, which conflicts were added, which
    files/refs/memberships moved) sufficient to reverse THAT merge → an **Unmerge** button restores
    the shadow to a standalone paper and undoes the moves.
  - **Flatten-on-re-merge:** if you merge into a base that already has a shadow (already-merged), the
    prior merge is **flattened first** (its shadow finalized/permanent, links fixed), then the new
    merge proceeds with a fresh shadow. So Unmerge always reverses only the **most recent** merge;
    older merges are permanent. Simple, stable, single-level — no chain to corrupt.
  - (A bounded full history is possible as a later enhancement if single-level proves limiting;
    starting simple per the owner's stability concern.)
- **Link** = keep BOTH entries and their own PDFs; record a **bidirectional "related / same work"**
  link so each shows + jumps to the other. No file move, no deletion. (Confirmed = what the owner
  wants.)
- Rename the actions (Merge / Link) + rework the review UI (order-swap control, per-action preview)
  + confirmations; the link-fixing (reference redirect) is shared infra used by merge + unmerge.
- **One remaining sub-decision for the owner:** OK to go with **single-level unmerge + flatten-on-
  re-merge** (my rec, for stability) rather than a full reversible merge-history? Proceeding with the
  rec unless told otherwise.

## Batch W — Makefile & Docker workflow robustness (owner 2026-07-06)
Owner pulled, rebuilt, `make up-all` errored, had to hand-fix (commits `45edd6e`, `5f4e874`). Root
cause: the frontend `node_modules` **named volume goes stale when `package-lock.json` changes**
(new deps `yaml`/`echarts`/`cytoscape`) — the rebuilt image is shadowed by the old volume → missing
deps → runtime errors. Owner's fix (bake node_modules + lock-hash into the image at
`/opt/paracord-frontend`, self-heal in the entrypoint on hash mismatch, add `make up-all`, chown
`dist`) is correct. Now make the WHOLE workflow robust out-of-the-box AND after changes:
- **Verify + solidify the frontend self-heal**: empty-volume first-run (copies baked node_modules,
  no npm ci), hash-match (no-op), hash-mismatch-with-baked (fast copy), hash-mismatch-no-baked
  (npm ci fallback); confirm the `.paracord-package-lock.sha256` marker logic is airtight.
- **Audit every named volume for ownership**: confirm backend `/app/storage` (done), frontend
  `node_modules` + `dist` (done); check the **agent** container + any other writable mount and chown
  in its entrypoint if it runs non-root.
- **Startup ordering / self-heal**: confirm compose `depends_on` conditions (api waits for Postgres
  *healthy* before migrations; worker waits for migrations-at-head + Redis) so a cold `up-all` orders
  itself; api entrypoint runs `alembic upgrade head` (done) — make sure a fresh DB comes up clean.
- **Recovery + clarity Make targets**: add `make rebuild` (build `--no-cache` + recreate) and
  `make fresh`/`reset` (down + drop the app volumes — postgres/node_modules/dist/library — then
  build + up, for a guaranteed clean slate; keep ollama models), and a dev `make smoke`/`doctor`
  that after `up-all` asserts each service is healthy (api `/health`, frontend serving, worker up,
  grobid/ollama when profiled). Document the canonical flow (`make init && make up-all`) + recovery
  (`make fresh`) in INSTALL/a runbook.
- **Prove it**: exercise a rebuild cycle — simulate a `package-lock.json` change → `make up-all`
  (rebuild) → confirm the frontend picks up deps with NO manual fix; and a clean `up-all` → all
  services healthy (smoke passes). Report the evidence.
