## Part A ‚Äî duplicate-PDF awareness ‚úÖ

- **Root cause (documented, full fix deferred by owner):** SHA-256 dedup (`storage.py`) makes an
  attached duplicate PDF reuse the original `File` row + add a new `FileWorkLink`. `File.status` is
  shared ‚Üí paper B shows "extracted" with no job. `extract_and_store` (extraction.py:249) resolves
  the work via an **unordered first `FileWorkLink`**, so every extraction writes to the file's
  original paper. Proper per-paper extraction of a shared PDF (thread `work_id` through
  enqueue‚Üíjob‚Üíextract; key the queue per (file,work); apply to the requesting work) is the deferred
  "better way".
- **Files changed:**
  - `backend/app/api/v1/endpoints/works.py` ‚Äî `WorkFileRead.also_in_count`; `_other_work_counts`
    (batched); `list_work_files` + `upload_work_file` populate it; attach skips the no-op
    re-extraction enqueue when a deduped file is already `extracted`.
  - `backend/app/services/duplicate_detection.py` ‚Äî new `_shared_file_candidates` (candidate_type
    `shared_file`, score 1.0) added to `find_work_candidates`; reads `FileWorkLink`.
  - `backend/openapi.json` regenerated (`also_in_count`). **No migration** (nothing changed the DB
    schema ‚Äî `also_in_count` is computed; `shared_file` is just a candidate_type value).
  - `frontend/src/api/client.ts` ‚Äî `WorkFile.also_in_count`.
  - `frontend/src/components/WorkDetail.svelte` ‚Äî clickable "duplicate PDF" badge (`.fdup`) +
    `findDuplicatePapers(sha)` (sets `pendingLibrarySearch` ‚Üí `#library`).
- **Tests:** `test_work_file_move_and_merge.py::test_files_report_also_in_count_for_shared_pdf`;
  `test_duplicate_detection.py::test_shared_file_candidate_flags_works_sharing_one_pdf` (+ its
  `db_session` fixture now creates `file_work_links`); `WorkDetail.dupfile.test.ts`.
- **Badge scope (owner decision):** count-only + hash search (no titles) ‚Äî avoids leaking titles of
  papers a LAN user may not see once access control lands.

## Part B ‚Äî dedup action semantics ‚úÖ

- **Owner-specified semantics:** "Ignore" = transient (drop now, reappear next scan, no permanent
  flag); "Keep separate" = permanent, reviewable, revocable.
- **Files changed:**
  - `backend/app/services/duplicate_resolution.py` ‚Äî `apply_duplicate_action` "ignore" now records an
    audit event then **deletes** the candidate (early return) instead of stamping status `ignored`.
    "keep_separate" unchanged (status `rejected`). Re-scan's `_upsert_candidate` re-creates the
    deleted pair as `open` (transient confirmed); a `rejected` row is reused untouched (permanent).
  - `backend/app/api/v1/endpoints/duplicates.py` ‚Äî `update_duplicate_candidate` captures the view
    before an "ignore" (row is deleted) and returns it as status `dismissed`.
  - `frontend/src/pages/DuplicatesPage.svelte` ‚Äî status filter "Rejected" ‚Üí "Kept separate"; dropped
    the now-unused "Ignored" option; button tooltips updated to the new semantics.
- **Tests:** `test_duplicates_api.py::test_ignore_action_deletes_candidate_but_keep_separate_persists`.
- **Note:** the direct `status="ignored"` PATCH path (not the `action="ignore"` path) still works and
  is still tested ‚Äî only the *action* semantics changed.
