## Goal

Bibliography references rarely link to a library paper even when the cited paper is genuinely local,
because titles differ by punctuation/dash/case (e.g. "KnowRob: A knowledge processing…" locally vs.
"KnowRob – A Knowledge Processing…" cited). The reference then shows "external" and its **Import**
creates a near-duplicate `Work`. Batch 12 adds a tolerant title(+year+author) matcher that surfaces
**"likely local"** candidates the owner confirms/rejects, plus the canonical-reference refactor that
makes matching run once per cited thing and apply to every citing work.

## Gotcha found in the field

- **`dedup_key` overflow on real data.** `Reference.dedup_key` is `String(512)`; the title-based key
  `title:<normalized_title>|<year>` was **uncapped**, so a long / mis-parsed reference title (whole
  citation lands in `title`) overflowed 512 chars → Postgres "value too long" → the **0059 backfill
  migration aborted on startup** (the API entrypoint runs `alembic upgrade head`), so the API
  container exited 1 on a machine with real library data. The parity tests use `create_all`, not the
  migration, so they never caught it. Fixed by capping the key in `reference_dedup_key`
  (`reference_links.py`) — the migration calls that same helper, so the backfill is fixed too.
  Regression: `test_reference_links.py::test_dedup_key_is_capped_to_fit_the_indexed_column`. (If a
  deployment still fails 0059, check the log for an `ix_references_normalized_title` btree
  "index row size exceeds maximum" — a >2704-byte normalized title — which would need a similar cap.)
